# Aiko Windows 桌宠设计规格

日期：2026-05-19

## 1. 背景

Aiko 是一个运行在 Windows 上的二次元少女型桌宠应用。它的目标不是成为 OpenClaw 一类大型通用 Agent，而是成为一个长期陪伴在用户桌面上的本地助手。

第一版采用平衡型 MVP：同时打通 Live2D 桌宠表现、本地助手操作、长期记忆三个最小闭环。

核心定位：

- 桌宠优先，而不是普通聊天窗口。
- 本地助手优先，而不是高权限全自动系统 Agent。
- 长期陪伴优先，而不是一次性问答工具。
- 权限可控优先，而不是让大模型直接操作系统。

## 2. 第一版目标

第一版需要完成以下闭环：

- Electron 透明置顶桌宠窗口。
- Web Live2D 渲染测试模型。
- 单角色二次元少女形象。
- 文字输入和气泡反馈。
- 大模型对话能力。
- 本地规则处理简单命令。
- 打开应用、打开网页、创建本地提醒。
- 首次确认、记住授权、撤销授权。
- SQLite 长期记忆主库。
- 回复前记忆召回。
- 回复后 Silent Memory Worker 异步整理记忆。
- 记忆面板查看和删除记忆。
- 本地日程提醒。
- 实时语音管线架构预留。

## 3. 非目标

第一版暂不做：

- 多角色系统。
- 角色商店。
- 复杂养成系统。
- 外部日历同步。
- 语音识别。
- 完整专业 TTS 模型接入。
- zero-shot voice cloning 实际推理。
- 高权限 Windows 操作。
- 跨应用 UI 自动化。
- 自动发送消息。
- 任意 shell 命令执行。
- 前台窗口、键鼠、应用使用时长监听。
- 删除或批量移动文件。

这些能力可以后续扩展，但不进入第一版主路径。

## 4. 技术路线

采用：

```text
Electron
+ Live2D Web 渲染
+ Node 主进程系统桥
+ 授权规则引擎
+ SQLite 本地数据层
+ GLM 大模型 API
+ Realtime Voice Pipeline 预留
```

选择 Electron 的原因：

- Windows 桌面窗口能力成熟。
- 支持透明、无边框、置顶、点击穿透等桌宠需求。
- WebGL 和 Live2D Web SDK 集成顺。
- Node 主进程适合承载本地系统桥。
- 后续可以通过独立进程扩展本地 TTS、脚本、Windows 能力。

## 6. UI 设计

详细人物和 UI 子规格见工作区根目录的 `人物UI.md`。

第一版 UI 分三层。

### 6.1 桌面常驻层

常驻层是用户最常看到的界面。

包含：

- Live2D 角色。
- 短气泡消息。
- 小型状态提示。
- 可隐藏菜单入口。

窗口要求：

- 透明背景。
- 无边框。
- 置顶。
- 可拖拽。
- 支持点击穿透开关。
- 支持缩放。
- 支持多显示器位置记忆。

设计原则：

- 角色是视觉中心。
- 默认不显示复杂按钮。
- 桌宠不应像一个完整工具窗口。

### 6.2 快速交互层

包含：

- 文字输入框。
- 主动建议气泡。
- 操作确认弹窗。

输入框触发方式：

- 点击角色。
- 快捷键。
- 托盘菜单。
- 右键菜单。

确认弹窗需要展示：

- 将执行的动作。
- 动作来源。
- 风险等级。
- 仅本次执行。
- 以后同类操作不再询问。
- 取消。

### 6.3 管理面板层

第一版包含 5 个面板：

- 聊天面板。
- 日程提醒面板。
- 记忆面板。
- 授权规则和应用白名单。
- 设置面板。

管理面板是工具界面，应清晰、紧凑、低装饰，不抢 Live2D 角色的主视觉。

## 7. Live2D 设计

第一版先用测试模型验证工程，接口按正式角色模型预留。

模型资源格式：

- `.model3.json`
- `.moc3`
- texture
- `physics3.json`
- expressions
- motions

基础表情：

- 默认。
- 微笑。
- 开心。
- 思考。
- 困惑。
- 认真。
- 提醒。
- 亲近。
- 担心。

基础动作：

- 待机呼吸。
- 眨眼。
- 视线移动。
- 打招呼。
- 点头。
- 摇头。
- 思考。
- 提醒。
- 成功反馈。
- 失败反馈。
- 被点击反馈。
- 拖拽反馈。

表情和动作由本地状态机驱动，大模型只能给出建议情绪，不直接控制底层动作。

## 8. 对话和意图路由

对话系统分为本地规则和大模型两部分。

### 8.1 本地规则路径

适合明确、低风险、结构简单的命令：

- 打开 VS Code。
- 打开浏览器。
- 打开指定网页。
- 30 分钟后提醒我。
- 明早 9 点提醒我。

流程：

```text
用户输入
-> 本地规则解析
-> 生成结构化意图
-> 权限检查
-> 执行或确认
```

### 8.2 大模型路径

适合模糊、复杂、需要规划或上下文理解的请求：

- 帮我安排今晚学习。
- 以后写代码前帮我准备环境。
- 根据我最近的计划提醒我。
- 解释一个复杂任务并拆步骤。

流程：

```text
用户输入
-> Memory Recall
-> LLM 理解和回复
-> 输出结构化候选动作
-> 本地权限层校验
-> 用户确认或执行
```

大模型不能直接执行系统操作。它只能输出建议动作或结构化意图，最终执行必须经过本地能力层和权限层。

## 9. 本地助手能力

第一版实现低风险能力。

### 9.1 打开应用

通过应用白名单执行。

每个应用配置：

- 应用名称。
- 可执行文件路径。
- 别名。
- 图标。
- 默认是否需要确认。

### 9.2 打开网页

支持打开指定 URL。

默认风险等级为低，但如果 URL 来自模型生成而不是用户明确输入，应要求确认。

### 9.3 打开文件夹或文件

第一版只支持打开用户配置过的安全路径，避免任意路径操作。

### 9.4 本地提醒

支持：

- 一次性提醒。
- 简单重复提醒。
- 暂停提醒。
- 删除提醒。
- 到点气泡提醒。

提醒存储在本地 SQLite，不接外部日历。

## 10. 授权规则

授权系统是安全边界。

### 10.1 风险等级

低风险：

- 打开应用。
- 打开网页。
- 打开已配置文件夹。
- 创建本地提醒。

中风险：

- 修改日程。
- 批量打开工作流。
- 运行用户配置脚本。

高风险：

- 删除文件。
- 移动大量文件。
- 自动发送消息。
- 自动提交表单。
- 修改系统设置。
- 执行任意 shell 命令。

第一版只实现低风险能力。中风险只预留结构，高风险明确不做。

### 10.2 确认策略

低风险操作：

- 首次执行需要确认。
- 用户可选择以后同类操作不再询问。
- 授权可撤销。

中风险操作：

- 后续版本再实现。
- 默认需要确认。

高风险操作：

- 第一版不实现。
- 后续也默认每次确认。

### 10.3 授权规则写入

授权规则不能由大模型直接写入。

流程：

```text
大模型或本地规则提出动作
-> 权限层判断风险
-> 用户确认
-> 本地权限层写入授权规则
```

## 11. 长期记忆系统

长期记忆是桌宠陪伴感的核心。

采用：

```text
SQLite = 运行时真实记忆库
Vector Index = 语义检索层
Markdown Export = 用户可读导出
```

### 11.1 记忆分类

第一版支持：

- 用户偏好。
- 称呼和关系。
- 常用软件。
- 习惯。
- 近期事件。
- 授权相关候选。
- 待确认记忆。

### 11.2 回复前记忆召回

每次用户输入后，主回复前运行 Memory Recall。

流程：

```text
用户消息
-> 检索相关长期记忆
-> 返回少量相关记忆
-> 注入主模型上下文
```

要求：

- 超时时间短。
- 找不到记忆不阻塞回复。
- 只返回和当前对话相关的记忆。

### 11.3 回复后静默整理

每轮对话结束后，Silent Memory Worker 异步运行。

流程：

```text
conversation_turn_completed
-> enqueue memory job
-> LLM 提取候选记忆
-> 本地规则分类风险
-> 写入、合并、待确认或丢弃
```

静默写入范围：

- 称呼偏好。
- 语气偏好。
- 常用软件。
- 低风险生活偏好。
- 角色互动偏好。
- 非敏感习惯线索。

必须确认范围：

- 日程。
- 提醒。
- 免确认授权规则。
- 敏感个人信息。
- 长期承诺。
- 涉及文件、系统、账号的自动化规则。

### 11.4 用户可控

记忆面板必须支持：

- 查看记忆。
- 删除记忆。
- 确认待处理记忆。
- 拒绝候选记忆。
- 导出 Markdown。

长期陪伴不能变成黑箱。

## 13. Realtime Voice Pipeline

语音是后续核心表现层，因此第一版必须按实时语音架构预留。

目标：

- 支持专业 TTS。
- 支持 zero-shot voice cloning。
- 支持低延迟流式播放。
- 支持 Live2D 口型同步。
- 支持后续本地 TTS Server。

### 13.1 管线

```text
LLM Streaming Text
-> Sentence Chunker
-> Streaming TTS Provider
-> Audio Queue
-> Audio Stream Player
-> Lip Sync Driver
-> Live2D Mouth Params
```

### 13.2 Provider 接口

接口必须支持流式和非流式两种模式。

```text
synthesizeStream({
  text,
  voiceProfileId,
  emotion,
  speed,
  format
}) -> AudioStream
```

```text
synthesizeOnce({
  text,
  voiceProfileId,
  emotion,
  speed,
  format
}) -> audioFile | audioBuffer
```

### 13.3 低延迟要求

需要支持：

- 大模型流式文本。
- 按短句或语义片段切分。
- 前一句播放时后一句合成。
- 播放队列。
- 用户打断。
- 取消当前 TTS 请求。
- 口型同步延迟控制。

### 13.4 zero-shot voice cloning 预留

后续接入专业 TTS 时需要：

- 参考音频路径。
- 声线 embedding 缓存。
- 情绪参数映射。
- 语速、音高、音量配置。
- 本地或远程推理服务地址。
- WebSocket 或 HTTP streaming。

第一版可以实现 Mock Provider，但接口不能按整段音频文件的思路写死。

## 14. 角色配置

第一版只做单角色，但保留结构化配置。

`character_profile` 包含：

- 角色名称。
- 用户称呼。
- 性格标签。
- 说话风格。
- 互动边界。
- Live2D 模型路径。
- 表情映射。
- 动作映射。
- 默认 voice profile。

不做多角色切换，但数据结构不应阻碍后续扩展。

## 15. 主动性设计

第一版主动性来自：

- 时间触发。
- 对话记忆触发。

主动行为示例：

- 到点提醒用户。
- 根据习惯建议打开常用应用。
- 根据近期事件提醒计划。
- 根据称呼和语气偏好调整回复。

主动行为原则：

- 不抢焦点。
- 低频、低打扰。
- 使用气泡提示。
- 用户拒绝后降低同类提醒频率。
- 涉及日程、授权、系统操作必须确认。

## 16. GLM API 接入边界

第一版将 GLM API 用于：

- 自然语言对话。
- 模糊意图理解。
- 复杂任务计划。
- 记忆候选提取。

不允许大模型直接：

- 执行系统命令。
- 写入授权规则。
- 创建高风险操作。
- 修改本地数据库中的敏感状态。

模型配置通过 `.env` 提供：

```text
GLM_BASE_URL
GLM_MODEL
GLM_API_KEY
```

API Key 不能写死在代码里。实现阶段从本地环境文件读取配置，并避免在日志、错误信息、界面或调试输出中显示密钥明文。

## 17. 错误处理

需要覆盖：

- 大模型 API 不可用。
- 本地规则解析失败。
- 应用路径不存在。
- 用户取消操作。
- 授权规则被撤销。
- 提醒触发失败。
- SQLite 读写失败。
- TTS Provider 不可用。
- Live2D 模型加载失败。

降级原则：

- Live2D 失败时显示基础错误状态。
- AI 失败时保留本地命令能力。
- TTS 失败时保留文字气泡。
- 记忆写入失败不阻塞主对话。
- 本地能力执行失败必须反馈给用户。

## 18. 测试策略

第一版测试重点：

- Electron 窗口透明、置顶、拖拽、点击穿透。
- Live2D 模型加载和动作触发。
- 本地规则意图解析。
- 授权确认和免确认规则。
- 打开应用和打开网页。
- 本地提醒创建和触发。
- Memory Recall 返回相关记忆。
- Silent Memory Worker 写入候选记忆。
- 记忆面板删除和确认。
- TTS 接口可 mock、可取消、可排队。

需要人工验证：

- 桌宠常驻体验是否低打扰。
- 气泡是否遮挡桌面。
- 确认弹窗是否足够清楚。
- 角色表情是否和系统状态匹配。

## 20. 设计结论

第一版 Aiko 应该做成一个小而完整的桌宠助手：

```text
看起来像桌宠
能完成低风险本地助手操作
能长期记住用户
能主动但克制地陪伴
能为实时专业语音留好架构
```

这比一开始追求“打穿 Windows”的大型 Agent 更稳，也更符合桌宠产品的节奏。高权限能力可以后续逐步增加，但第一版必须先把角色、记忆、授权和低风险本地能力做扎实。
