# 人物 UI 设计草案

## 1. 定位

本桌宠是一个应用于 Windows 的单角色二次元少女型桌宠。她不是普通聊天窗口，也不是大型系统 Agent，而是一个长期陪伴在用户桌面上的本地助手角色。

第一版目标：

- 以 Live2D 少女角色作为桌面常驻主体。
- 支持文字对话、主动提醒、日程和本地操作建议。
- 通过长期记忆形成稳定陪伴感。
- 通过确认和授权机制安全执行 Windows 本地操作。
- 语音能力预留 TTS Provider，后续可接专业 TTS 和 zero-shot voice cloning。

核心体验应是：

> 用户平时看到的是一个有生命感的少女桌宠；需要帮助时，她能自然地回应、提醒、建议并执行已授权的本地操作。

## 2. 角色方向

### 2.1 基础人设

角色类型：二次元少女型桌宠助手

建议气质：

- 温柔但不软弱。
- 亲近但不过度打扰。
- 有轻微少女感和陪伴感。
- 面对工作任务时可靠、认真。
- 面对用户疲惫或低落时语气柔和。
- 面对系统操作时表达清晰、可确认。

推荐关键词：

- 陪伴
- 细心
- 稍微元气
- 可靠
- 轻度俏皮
- 低打扰
- 长期熟悉感

不建议第一版做成：

- 强傲娇，容易影响工具确认的清晰度。
- 过度卖萌，容易降低助手可信度。
- 冷淡秘书型，陪伴感不足。
- 复杂剧情型，容易分散 MVP 目标。

### 2.2 角色和用户的关系

第一版关系定位：

- 她是用户的桌面陪伴者和本地助手。
- 她会记住用户明确或低风险表达出的偏好。
- 她可以主动提醒，但需要尊重用户节奏。
- 她不会自作主张执行高风险操作。

关系变化可以先不做数值系统，但语气可以随记忆逐渐稳定：

- 初期：礼貌、轻微亲近。
- 熟悉后：使用用户偏好的称呼和语气。
- 长期使用后：能自然引用用户习惯、常用软件、近期计划。

## 3. 视觉设计

### 3.1 整体风格

视觉方向：

- 二次元 Live2D 少女角色。
- 桌面透明背景。
- UI 面板轻量、清晰、半透明。
- 功能界面弱化存在感，角色本体优先。

桌宠平时不应像一个完整应用窗口，而应像一个自然存在于桌面的角色。

### 3.2 角色造型建议

第一版正式角色建议采用适合桌面常驻的 3/4 视角半身或接近全身立绘。

建议特征：

- 轮廓清晰，缩小时仍容易辨认。
- 服装有识别点，但不要过度复杂。
- 面部表情可读性强。
- 手部和上半身适合做打招呼、思考、提醒动作。
- 颜色不要和 Windows 桌面常见背景过度混淆。

可选主题方向：

- 数字精灵型：轻微科技感，适合本地助手。
- 居家陪伴型：柔和、温暖，适合长期陪伴。
- 学习搭档型：干净、清爽，适合提醒和日程。

第一版推荐：

> 居家陪伴型 + 轻微数字助手元素。

这样既符合桌宠陪伴定位，也能自然承接打开软件、提醒、记忆、日程等功能。

### 3.3 Live2D 模型资源要求

正式模型建议包含：

- `.model3.json`
- `.moc3`
- texture 纹理文件
- `physics3.json`
- expressions 表情文件
- motions 动作文件

PSD 分层阶段应提前考虑：

- 头发前后层。
- 眼睛、瞳孔、高光、眼皮。
- 眉毛。
- 嘴型。
- 脸红。
- 手臂和手部。
- 衣服主要部件。
- 装饰物。
- 可动配饰。

如果后续要接 TTS 和口型同步，嘴型参数需要预留清楚。

## 4. 表情系统

### 4.1 第一版基础表情

建议至少准备以下表情：

| 表情 | 使用场景 |
| --- | --- |
| 默认 | 常驻待机、普通对话 |
| 开心 | 操作成功、用户正向反馈 |
| 微笑 | 日常陪伴、温和回应 |
| 思考 | 理解复杂意图、规划日程 |
| 困惑 | 不确定用户意图、需要澄清 |
| 认真 | 权限确认、重要提醒 |
| 提醒 | 日程到点、主动建议 |
| 害羞或亲近 | 关系型互动、称呼偏好确认 |
| 疲惫或低电量 | 夜间、长时间待机、可选状态 |
| 担心 | 用户表达压力或负面情绪 |

第一版不需要做太多夸张表情。表情重点是可读、稳定、和功能状态匹配。

### 4.2 表情触发规则

示例映射：

| 系统状态 | 表情 |
| --- | --- |
| 普通回复 | 默认或微笑 |
| 本地命令执行成功 | 开心 |
| 命令执行失败 | 困惑或担心 |
| 等待用户确认 | 认真 |
| 到点提醒 | 提醒 |
| 模型正在思考 | 思考 |
| 用户情绪低落 | 担心 |
| 用户表达亲近偏好 | 害羞或亲近 |

表情触发应由本地状态机控制，不应完全依赖大模型自由决定。

## 5. 动作系统

### 5.1 第一版基础动作

建议准备：

- 待机呼吸
- 自动眨眼
- 轻微视线移动
- 打招呼
- 点头
- 摇头
- 思考动作
- 提醒动作
- 操作成功动作
- 操作失败动作
- 被点击反馈
- 拖拽时反馈

### 5.2 动作触发场景

| 场景 | 动作 |
| --- | --- |
| 应用启动 | 打招呼 |
| 用户点击角色 | 被点击反馈 |
| 用户拖拽角色 | 拖拽反馈或保持待机 |
| 用户发送消息 | 轻微关注动作 |
| 等待模型回复 | 思考动作 |
| 创建提醒成功 | 点头或开心动作 |
| 打开应用成功 | 操作成功动作 |
| 执行失败 | 困惑动作 |
| 主动提醒 | 提醒动作 |

动作频率需要克制。桌宠常驻时，过多动作会打扰用户。

## 6. 桌面常驻 UI

### 6.1 常驻层

常驻层只包含必要元素：

- Live2D 角色。
- 简短聊天气泡。
- 小型状态提示。
- 可隐藏的小菜单入口。

常驻窗口要求：

- 透明背景。
- 无边框。
- 可置顶。
- 可拖拽。
- 支持点击穿透开关。
- 支持缩放。
- 支持多显示器位置记忆。

默认状态下，不应显示复杂控制按钮。

### 6.2 气泡设计

气泡用于短消息、主动建议和提醒。

建议规则：

- 单条气泡尽量短。
- 重要操作不要只依赖气泡，必须进入确认弹窗。
- 主动建议可使用轻量气泡。
- 长回复可以进入聊天面板，不在桌面上占用太多空间。

气泡类型：

- 普通对话气泡。
- 主动建议气泡。
- 日程提醒气泡。
- 操作结果气泡。
- 记忆确认气泡。

示例：

```text
要不要现在打开 VS Code？
```

```text
我记住啦，以后打开项目时会先帮你准备这些工具。
```

```text
你的日语学习时间到了，要开始吗？
```

### 6.3 输入入口

第一版采用文字输入。

触发方式：

- 点击角色。
- 快捷键。
- 托盘菜单。
- 右键菜单。

输入框要求：

- 不常驻占屏。
- 唤出后聚焦输入。
- 支持发送自然语言。
- 支持取消。
- 支持简单历史记录。

语音输入第一版不做，TTS 输出只预留接口。

## 7. 功能面板 UI

第一版需要 5 个核心界面。

### 7.1 聊天面板

用途：

- 长文本交流。
- 查看最近对话。
- 输入复杂任务。
- 展示模型计划和解释。

要求：

- 不取代桌宠本体。
- 从桌宠旁边或独立小窗打开。
- 支持文本输入。
- 支持执行计划确认。

### 7.2 确认弹窗

用途：

- 执行 Windows 本地操作前确认。
- 创建日程或提醒前确认。
- 写入高风险记忆前确认。
- 创建免确认授权规则前确认。

确认弹窗必须清楚说明：

- 将执行什么。
- 为什么执行。
- 风险等级。
- 是否允许以后免确认。

示例：

```text
要打开应用：Visual Studio Code

来源：你刚才说“打开 VS Code”
风险：低

[仅这一次] [以后打开 VS Code 不再询问] [取消]
```

### 7.3 日程提醒面板

用途：

- 查看本地提醒。
- 创建提醒。
- 删除提醒。
- 暂停提醒。
- 查看近期日程。

第一版只做本地日程，不接 Outlook、Google Calendar 或 Windows Calendar。

### 7.4 记忆面板

用途：

- 查看长期记忆。
- 删除错误记忆。
- 确认待处理记忆。
- 导出 Markdown。
- 查看最近记忆更新。

记忆分类：

- 用户偏好。
- 称呼和关系。
- 常用软件。
- 习惯。
- 近期事件。
- 待确认记忆。

记忆面板必须让用户知道桌宠“记住了什么”。长期陪伴不能变成黑箱。

### 7.5 设置面板

用途：

- Live2D 模型路径。
- 角色配置。
- OpenAI API 配置。
- TTS Provider 配置预留。
- 本地应用白名单。
- 授权规则。
- 窗口行为。
- 开机启动。

设置面板应偏工具化，清楚、紧凑，不做营销式页面。

## 8. 记忆和陪伴 UI

### 8.1 静默记忆写入

每轮对话结束后，Silent Memory Worker 在后台整理候选记忆。

静默写入范围：

- 称呼偏好。
- 语气偏好。
- 常用软件。
- 低风险生活偏好。
- 角色互动偏好。
- 非敏感习惯线索。

必须确认范围：

- 日程。
- 提醒。
- 免确认授权规则。
- 敏感个人信息。
- 长期承诺。
- 涉及文件、系统、账号的自动化规则。

### 8.2 记忆反馈方式

低风险记忆不需要每次打断用户，但可以用轻提示表达：

```text
我会记住你喜欢这个称呼。
```

高风险记忆需要确认：

```text
你是想让我以后每天 21:00 提醒你学习日语吗？
```

记忆面板里要能看到：

- 记忆内容。
- 来源时间。
- 分类。
- 是否确认。
- 是否用于主动提醒。
- 删除按钮。

## 9. 授权与安全 UI

### 9.1 授权等级

低风险：

- 打开应用。
- 打开网页。
- 打开指定文件夹或文件。
- 创建本地提醒。

中风险：

- 修改日程。
- 运行用户配置的脚本。
- 批量打开工作流。

高风险：

- 删除文件。
- 移动大量文件。
- 自动发送消息。
- 自动提交表单。
- 修改系统设置。
- 执行任意 shell 命令。

第一版只实现低风险能力，部分中风险能力可作为配置预留。

### 9.2 免确认规则

当用户确认某个低风险操作时，可以选择：

- 仅这一次。
- 以后同类操作不再询问。
- 取消。

免确认规则必须可以在设置面板撤销。

授权规则不应由大模型直接写入。模型只能提出建议，最终由本地权限系统保存。

## 10. 主动性 UI

第一版主动性来自：

- 时间触发。
- 对话记忆触发。

不做：

- 前台窗口监听。
- 键鼠行为监控。
- 应用使用时长监控。

主动提醒原则：

- 使用气泡，不抢焦点。
- 高频提醒可被静音。
- 用户拒绝后应降低同类提醒频率。
- 重要提醒可以带确认按钮。

示例：

```text
现在是你常用的学习时间，要不要开始？
```

```text
你昨天说今天想整理 Aiko 项目，要我帮你打开 VS Code 吗？
```

## 11. 角色反馈和系统状态映射

| 系统状态 | UI 表现 | 角色表现 |
| --- | --- | --- |
| 待机 | 无气泡或短状态 | 呼吸、眨眼 |
| 用户唤醒 | 输入框出现 | 看向用户 |
| 本地规则识别成功 | 确认弹窗或直接执行 | 认真 |
| 大模型思考中 | 小型加载状态 | 思考 |
| 等待确认 | 确认弹窗 | 认真、等待 |
| 操作成功 | 成功提示 | 开心、点头 |
| 操作失败 | 失败提示 | 困惑 |
| 到点提醒 | 提醒气泡 | 提醒动作 |
| 写入记忆 | 轻提示或记忆面板更新 | 微笑 |

## 12. 第一版不做的内容

为了保证第一版能落地，暂不做：

- 多角色系统。
- 角色商店。
- 复杂养成系统。
- 语音识别。
- 完整 TTS 接入。
- 跨应用 UI 自动化。
- 高权限 Windows 操作。
- 外部日历同步。
- 自动发送消息。
- 任意 shell 命令执行。

## 13. 后续扩展方向

可在第一版稳定后扩展：

- 专业 TTS Provider。
- zero-shot voice cloning 声线。
- Live2D 口型同步。
- 多套服装或表情包。
- 语音输入。
- 外部日历同步。
- 工作流自动化。
- 更复杂的记忆时间线。
- 多角色系统。

## 14. 当前设计结论

第一版人物 UI 的核心不是“做一个可爱的聊天窗口”，而是：

> 用 Live2D 少女角色承载长期陪伴感，用轻量桌面 UI 承载低打扰互动，用结构化记忆和授权系统承载可靠的本地助手能力。

这个方向可以先用测试 Live2D 模型验证工程，再替换为正式角色模型。正式角色制作时，应优先保证表情、动作和功能状态之间的映射，而不是只追求立绘复杂度。
